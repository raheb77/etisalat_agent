# Internal CSR Decision Support System Architecture

## 1. نظرة عامة على النظام
هذا النظام مخصص لدعم اتخاذ القرار فقط.
لا ينفذ أي تغييرات على الحسابات أو الخدمات.
لا يقوم بتخزين معلومات تعريف شخصية (PII).

## 2. طبقات المعمارية
- **CSR UI**: واجهة الوكيل لتجميع مدخلات العميل وعرض التوصيات بدون تنفيذ مباشر.
- **API Gateway**: بوابة موحدة لاستقبال الطلبات، تطبيق سياسات الدخول، وتوجيهها للطبقات الداخلية.
- **PII Sanitization Layer**: طبقة تنظيف وإزالة أي بيانات شخصية قبل المعالجة أو التسجيل.
- **Router (Category Classifier)**: مصنف لتحديد الفئة الموضوعية للطلب وربطها بالسياسات المناسبة.
- **Policy Engine**: محرك قواعد يطبق السياسات التشغيلية ويقيّد الاستجابات ضمن النطاق المسموح.
- **Structured Fact Resolver**: مفسر للحقائق المنظمة من `knowledge/facts` لعرض القيم المؤكدة فقط.
- **Vector Retrieval Layer**: استرجاع سياقي للمستندات النصية ذات الصلة من `docs/`.
- **LLM Adapter**: طبقة اتصال بالنموذج لتوليد ردود ضمن القيود والسياسات.
- **Confidence Engine**: تقدير مستوى الثقة لتحديد مسار الاستجابة أو التحويل.
- **Handoff Engine**: آلية تحويل للحالات منخفضة الثقة أو خارج النطاق.
- **Audit & Observability**: تسجيل وتشخيص تشغيلي دون تضمين PII.

## 3. تدفق الطلب من البداية للنهاية
1. يدخل وكيل CSR مدخلات العميل عبر الواجهة.
2. يمر الطلب عبر بوابة API للتحقق من الصلاحيات.
3. تُزال البيانات الشخصية عبر طبقة تنقية PII.
4. يصنف Router الطلب لفئة موضوعية.
5. يطبق Policy Engine القيود والسياسات الملائمة للفئة.
6. يستدعي Structured Fact Resolver الحقائق الرقمية المؤكدة فقط.
7. يسترجع Vector Retrieval Layer سياقاً نصياً داعماً من الوثائق.
8. يولد LLM Adapter صياغة الرد ضمن السياسات.
9. يقيّم Confidence Engine الثقة في الإجابة.
10. إذا كانت الثقة منخفضة أو خارج النطاق، يفعّل Handoff Engine التحويل.
11. تُسجل العملية في Audit & Observability دون PII.

## 4. حدود النظام (Non-goals)
- لا وصول للكتابة إلى أنظمة CRM.
- لا تنفيذ آلي لإلغاء الخدمات.
- لا تفسير قانوني أو إصدار أحكام تنظيمية.
- لا اتخاذ قرارات تحقيق احتيال.

## 5. نموذج النشر لفريق CSR مكوّن من 20 وكيل
- نشر داخل الشبكة الداخلية.
- المصادقة مطلوبة لكل جلسة.
- تسجيل تشغيلي بدون PII.
- وجود Kill-switch لتعطيل النظام عند الحاجة التشغيلية.

## 6. القيود الرقمية
- لا تُذكر أي أرقام إلا إذا كانت مؤكدة وموجودة في `knowledge/facts/`.
- أي رقم غير واضح يُوسم "Unverified" ولا يُضاف إلى `knowledge/facts`.
